# Task Manager — AWS Lambda + DynamoDB + Vite

A full-stack task management application built with a serverless backend (AWS Lambda + DynamoDB via Terraform) and a React/TypeScript frontend deployed on Vercel with CI/CD via GitHub integration.

**Live:** https://frontend-jaapaz.vercel.app  
**API:** https://7if1x7cbrl.execute-api.us-east-1.amazonaws.com/tasks

---

## Features Implemented

### 1. Serverless REST API (Node.js 20.x — Lambda)
Four endpoints handled by a single Lambda function using internal routing on `rawPath` and HTTP method:

| Method | Path | Description |
|---|---|---|
| `GET` | `/tasks` | List all tasks (DynamoDB Scan) |
| `POST` | `/tasks` | Create task — generates UUID v4, timestamps |
| `PUT` | `/tasks/{id}` | Partial update — title, description, completed |
| `DELETE` | `/tasks/{id}` | Hard delete by primary key |

All responses include `Content-Type: application/json` and `Access-Control-Allow-Origin: *`. Errors return `{ "error": "message" }` with the appropriate HTTP code.

### 2. DynamoDB Data Model
Table `tasks` with on-demand billing (`PAY_PER_REQUEST`):

| Attribute | Type | Notes |
|---|---|---|
| `id` | String (PK) | UUID v4 generated in Lambda |
| `title` | String | Required |
| `description` | String | Optional, defaults to `""` |
| `completed` | Boolean | Defaults to `false` |
| `createdAt` | String | ISO 8601 |
| `updatedAt` | String | ISO 8601, updated on every PUT |

### 3. React Frontend (Vite + TypeScript + Tailwind CSS)
- List tasks with visual state (completed / pending)
- Add task — title required, description optional
- Toggle completed / pending with animated circular checkbox
- Inline edit — title and description, `Enter` to save, `Escape` to cancel
- Delete with two-click confirmation (auto-resets after 3 s)
- Filters — All / Pending / Completed with live counts
- Loading spinner and inline error banner
- Responsive mobile-first layout

## Architecture

```
Browser (Vite SPA)
      │  fetch()
      ▼
API Gateway HTTP API (v2)
      │  AWS_PROXY  payload_format_version=2.0
      ▼
AWS Lambda  task_manager  (nodejs20.x)
      │  @aws-sdk/client-dynamodb  v3
      ▼
DynamoDB  tasks  (PAY_PER_REQUEST)
```

**Infrastructure as Code** — all AWS resources are managed by Terraform (`terraform/main.tf`). The Lambda ZIP is built automatically via the `archive_file` data source; no manual packaging step.

**Custom Hook pattern** — `useTasks` encapsulates all API calls and optimistic state mutations. Components only call `createTask`, `updateTask`, `deleteTask` — never `fetch` directly.

**Single-table design** — a DynamoDB Scan is acceptable here because task lists are per-user and small. No GSIs are required.

---

## How It Works

On mount, `useTasks` calls `GET /tasks` and stores the result sorted by `createdAt` descending. Every mutation (create / update / delete) calls the Lambda and immediately patches local state — no full re-fetch. The Lambda uses DynamoDB's `UpdateExpression` with `ReturnValues: "ALL_NEW"` so the updated item is returned in a single round-trip.

```typescript
// hooks/useTasks.ts — optimistic update on toggle
const updateTask = async (id: string, payload: UpdateTaskPayload) => {
  const updated = await api.update(id, payload)
  setTasks(prev => prev.map(t => (t.id === id ? updated : t)))
}
```

```javascript
// lambda/index.mjs — partial update expression built dynamically
if (title !== undefined)     { exprs.push("#t = :t"); names["#t"] = "title"; values[":t"] = { S: title } }
if (completed !== undefined) { exprs.push("#c = :c"); names["#c"] = "completed"; values[":c"] = { BOOL: completed } }
```

## Getting Started

### Prerequisites
- [Terraform](https://developer.hashicorp.com/terraform/install) ≥ 1.5
- AWS CLI configured with profile `jaap-2026`
- Node.js ≥ 20.x
- [Vercel CLI](https://vercel.com/docs/cli) (optional, for manual deploy)

### 1. Deploy infrastructure

```bash
cd terraform
terraform init
AWS_PROFILE=jaap-2026 terraform apply -auto-approve
# Copy the lambda_url output value
```

### 2. Run the frontend locally

```bash
cd frontend
echo "VITE_API_URL=https://<lambda_url>" > .env.local
npm install
npm run dev
# → http://localhost:5173
```

### 3. Test the API directly

```bash
LAMBDA_URL=https://<lambda_url>

# Create
curl -X POST $LAMBDA_URL/tasks \
  -H "Content-Type: application/json" \
  -d '{"title":"Buy milk","description":"Full fat"}'

# List
curl $LAMBDA_URL/tasks

# Complete
curl -X PUT $LAMBDA_URL/tasks/<id> \
  -H "Content-Type: application/json" \
  -d '{"completed":true}'

# Delete
curl -X DELETE $LAMBDA_URL/tasks/<id>
```

### 4. Deploy frontend to Vercel

#### Option A — CLI (manual deploy)

```bash
npm install -g vercel
cd frontend
vercel whoami                  # authenticate if needed
echo "https://<lambda_url>" | vercel env add VITE_API_URL production
vercel --prod --yes
```

#### Option B — Git integration (CI/CD, recommended)

1. Go to **[vercel.com/jaapaz/frontend/settings/git](https://vercel.com/jaapaz/frontend/settings/git)**
2. Click **"Connect Git Repository"** → GitHub
3. Authorize the **Vercel GitHub App** on the `Jorgeaapaz` account (select only `MISEIA_1-5-50-aws-lambda-vite`)
4. Set **Root Directory:** `frontend` · **Production branch:** `main`
5. Click **"Connect"**

Once connected, every `git push origin main` deploys automatically to production. Pull Requests get a unique preview URL and a status check on GitHub.
